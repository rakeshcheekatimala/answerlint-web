# Changelog

All notable changes to this project will be documented here when a release is published.

This project follows the spirit of [Keep a Changelog](https://keepachangelog.com/en/1.1.0/): release notes should be written for humans, grouped by impact, and should not be a raw git log.

## Release Notes Workflow

- Pull requests must have a human-readable title and a short `Release Notes` section.
- Pull requests should use one release label: `type: feature`, `type: fix`, `type: ux`, `type: docs`, or `type: infra`.
- Use `no-release-note` for internal changes that should not appear in public release notes.
- When a pull request is merged into `master`, GitHub Actions updates a draft GitHub Release.
- When you are ready to publish a release, review the draft, adjust wording if needed, publish it, and copy the final summary into this changelog.
